SDD · Spec-Driven Development

Cómo funciona SDD en este proyecto

SDD organiza el trabajo en cambios pequeños y trazables, conectando intención, especificación, implementación y verificación antes de cerrar cada cambio.

¿Qué es SDD?

Spec-Driven Development (SDD) es una forma de trabajar donde primero se define qué se quiere cambiar y cómo se valida, y después se implementa.

  • Evita cambios improvisados sin contexto.
  • Mejora la trazabilidad de decisiones técnicas.
  • Facilita revisar y retomar trabajo entre sesiones.

Comando inicial: sdd:new

Cuando propones una idea nueva, empieza con un identificador de cambio claro:

sdd:new agregar-explicacion-sdd-html

Eso abre el flujo para documentar el cambio dentro de openspec/changes/<cambio>/.

Estructura base de SDD

openspec/ ├── config.yaml ├── specs/ └── changes/ ├── archive/ └── <cambio>/ ├── proposal.md ├── design.md ├── tasks.md └── specs/

Flujo recomendado (de punta a punta)

  1. Explore: entender alcance, riesgos y módulos afectados.
  2. Propose: redactar proposal.md con intención y plan.
  3. Spec: definir requisitos y escenarios (Given/When/Then).
  4. Design: documentar decisiones técnicas y tradeoffs.
  5. Tasks: dividir implementación en tareas concretas.
  6. Apply: implementar código siguiendo tasks y specs.
  7. Verify: validar que todo cumpla lo especificado.
  8. Archive: cerrar el cambio y moverlo a archive/.

Ejemplo rápido para esta app

# 1) Crear cambio sdd:new agregar-html-sdd # 2) Definir intención y alcance sdd:propose agregar-html-sdd # 3) Escribir requisitos/escenarios sdd:spec agregar-html-sdd # 4) Implementar y verificar sdd:apply agregar-html-sdd sdd:verify agregar-html-sdd

Buenas prácticas

  • Nombrar cambios en infinitivo o acción clara (agregar-*, corregir-*).
  • Mantener los escenarios de prueba en lenguaje de negocio.
  • No mezclar múltiples objetivos grandes en un mismo cambio.
  • Cerrar cada cambio con verificación y archivo para no dejar deuda documental.
Tip: Si el cambio impacta UI, conserva el estilo existente del frontend para mantener consistencia visual.
SDD ayuda a construir cambios claros, verificables y mantenibles en equipo.
