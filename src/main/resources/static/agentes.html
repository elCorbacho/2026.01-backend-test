Agentes

Estructura de agentes en este proyecto

El repositorio mezcla habilidades específicas para Spring Boot y diseño de interfaces. Esta página describe cómo están organizados los agentes, sus roles y referencias clave.

Directorio de skills

Las habilidades vivas están en .agents/skills/. Cada carpeta contiene el archivo principal SKILL.md, referencias adicionales y recursos complementarios.

  • spring-boot-engineer: especialización senior orientada a microservicios, WebFlux, Spring Security y Spring Data.
  • java-spring-boot: enfoque productivo para APIs REST, seguridad y monitoreo; incluye plantillas y validadores.
  • frontend-design: guía visual para generar interfaces con diseño deliberado.

Al invocar un skill, el agente adopta el rol descrito en su SKILL, con restricciones (MUST DO / MUST NOT DO) y salidas prediseñadas.

spring-boot-engineer

Carpeta: .agents/skills/spring-boot-engineer/

Actúa como ingeniero senior. Insiste en capas limpias, pruebas, transacciones y seguridad. Tipos:

  • Referencia web: references/web.md
  • Acceso a datos: references/data.md
  • Seguridad: references/security.md
  • Testing: references/testing.md
  • Cloud: references/cloud.md

java-spring-boot

Carpeta: .agents/skills/java-spring-boot/

Orientado a productividad: ofrece ejemplos de `SecurityConfig`, `GlobalExceptionHandler`, plantillas YAML y patrones comunes.

Admite parámetros (`spring_version`, `module`) y contiene scripts de validación y assets para generar respuestas consistentes.

frontend-design

Carpeta: .agents/skills/frontend-design/

Guía para crear interfaces deliberadas. Incluye la licencia y explica cómo generar layouts, tipografía y experiencia visual.

Flujo de un agente

1. Reconocer la necesidad: al detectar una tarea que cae en Spring Boot o frontend, se invoca el skill correspondiente.
2. Aplicar restricciones: seguir los MUST DO/MUST NOT DO del SKILL, usar referencias para cada subtema y documentar decisiones.
3. Responder con plantilla: los SKILL incluyen output templates (entidades, servicios, DTOs, tests) que facilitan respuestas coherentes.
4. Registrar memoria: cada cambio importante se acompaña de un `mem_save`/`mem_session_summary` para mantener el contexto.
