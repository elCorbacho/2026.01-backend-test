Spec-Driven Development

¿Qué es OpenSpec?

OpenSpec es una metodología de desarrollo impulsada por especificaciones donde se define qué, cómo y cuándo antes de implementar.

Definición

Spec-Driven Development (SDD) es una forma de trabajar donde primero se documenta qué se quiere cambiar, cómo se valida, y después se implementa. Esto evita cambios improvisados y mejora la trazabilidad de decisiones técnicas.

Beneficios clave:
  • Claridad: todos entienden exactamente qué se va a hacer antes de empezar.
  • Trazabilidad: cada cambio queda documentado con su intención, especificación y diseño.
  • Verificabilidad: puedes validar que la implementación cumple con lo especificado.
  • Retomabilidad: si te interrumpen, puedes retomar el trabajo meses después sin confusión.
Flujo recomendado (8 pasos)
  1. Explore: entiende el alcance, riesgos y módulos afectados.
  2. Propose: redacta proposal.md con intención y plan.
  3. Spec: define requisitos y escenarios (Given/When/Then).
  4. Design: documenta decisiones técnicas y trade-offs.
  5. Tasks: divide la implementación en tareas concretas.
  6. Apply: implementa el código siguiendo tasks y specs.
  7. Verify: valida que todo cumpla lo especificado.
  8. Archive: cierra el cambio y muévelo a archive/.
Comando inicial

Inicia un nuevo cambio con un nombre descriptivo:

terminal 
sdd-propose "agregar-vista-openspec-y-skills-teams"
Estructura de directorios

Los cambios viven dentro de openspec/changes/:

openspec-tree 
openspec/changes/mi-cambio/
├── proposal.md
├── specs/
│   └── {domain}/spec.md
├── design.md
└── tasks.md
Buenas prácticas
  • Nombres descriptivos: usa infinitivo o acciones claras (agregar-*, corregir-*, refactorizar-*).
  • Cambios pequeños: mantén cada cambio enfocado en un objetivo claro. Evita mezclar múltiples features grandes.
  • Escenarios testeable: cada requisito debe tener al menos un escenario Given/When/Then que pueda automatizarse.
  • Cierre completo: no dejes cambios a mitad de camino. Verifica y archiva para evitar deuda documental.
Skills Agent Teams

¿Qué son Skills Agent Teams?

Equipos de agentes especializados que trabajan en colaboración para resolver tareas complejas del proyecto.

Definición

Skills Agent Teams son agentes IA especializados, cada uno con un rol específico y acceso a herramientas particulares. Trabajan de forma autónoma en tareas complejas, aplicando sus habilidades especializadas de manera coordinada.

Características:
  • Especializados: cada agente tiene experticia en un dominio (backend, frontend, testing, etc.).
  • Autónomos: pueden trabajar sin supervisión constante en tareas bien definidas.
  • Herramientas específicas: cada agente tiene acceso a tools relevantes para su dominio.
  • Colaborativos: pueden coordinarse y comunicarse con otros agentes cuando es necesario.
Agentes disponibles en el proyecto
general-purpose

Manejo general de tareas, investigación compleja, búsqueda de código.

Explore

Exploración rápida de codebases, búsqueda de patrones, respuestas sobre arquitectura.

Plan

Diseño de arquitectura, planes de implementación, análisis de trade-offs.

frontend-ui-ux-expert

Diseño de componentes, UX/accesibilidad, sistemas de diseño.

Cómo funcionan
Flujo de invocación:
  1. Descripción: describes la tarea que necesitas (feature, bug fix, investigación).
  2. Selección: el sistema elige el agente o equipo más apropiado para el trabajo.
  3. Análisis: el agente analiza el contexto, lee código relevante, identifica patrones.
  4. Ejecución: realiza la tarea usando sus herramientas especializadas (búsqueda, edición, testing, etc.).
  5. Reporte: documenta qué se hizo, decisiones tomadas, y aprende para futuras tareas.
Caso de uso 1: Refactor Backend

El agente de Plan diseña la arquitectura, el general-purpose busca patrones, y el Explore valida que la refactorización no rompe nada.

Caso de uso 2: Nueva UI

El experto frontend diseña componentes accesibles, propone patrones, y el general-purpose integra con la API.

Caso de uso 3: Bug Investigation

El Explore busca el código afectado, el general-purpose analiza logs, identifica la raíz del problema.

Caso de uso 4: Architecture Decision

El Plan analiza trade-offs, el Explore valida con el codebase actual, documentan la decisión.
